# hashcracker

A small, well-commented **educational** hash cracker for learning how password
storage and cryptographic hash functions actually work. It supports **MD5**,
**SHA-1**, and **SHA-256**, with two attack modes (dictionary and brute-force),
algorithm auto-detection, a live progress/rate display, and multiprocessing
across CPU cores.

---

## ⚠️ Ethical use — please read first

**This is a learning tool.** It exists so you can understand *why* fast,
unsalted hashes are weak and *how* real password-cracking works — knowledge
that makes you better at defending systems.

Only ever run it against:

- **hashes you generated yourself** (use the included `scripts/make_hashes.py`), or
- **CTF challenge hashes**, or
- systems you have **explicit written authorisation** to test.

Cracking hashes you don't own or aren't authorised to test is unauthorised
access and is illegal in most jurisdictions. Don't. The whole point here is to
learn the concepts, not to attack anyone.

---

## What it does

- **Dictionary attack** — hashes each line of a wordlist and compares to the target.
- **Brute-force attack** — generates every combination from a character set up
  to a maximum length using `itertools.product`.
- **Algorithms** — MD5, SHA-1, SHA-256 via `hashlib.new()`.
- **Auto-detection** — infers the algorithm from the hash length (32/40/64 hex
  chars → MD5/SHA-1/SHA-256), overridable with `--algo`.
- **Progress** — a live `tried | hashes-per-second | elapsed | % | ETA` line.
- **Parallelism** — splits work across all CPU cores with `multiprocessing`.
- **Clean errors** — friendly messages and sensible exit codes, never a traceback.

Zero third-party dependencies — everything is Python standard library.

## Generating your own test hashes

You need something to crack that you already know the answer to. `make_hashes.py`
does exactly that:

```bash
# MD5 of "password"
python scripts/make_hashes.py password
# -> 5f4dcc3b5aa765d61d8327deb882cf99

# Choose the algorithm, and show the plaintext next to the digest
python scripts/make_hashes.py letmein --algo sha256 --show
# -> letmein -> 1c8bfe8f801d79745c4631d09fff36c82aa37fc4cce4fc946683d7b336b63032

# Hash several values at once
python scripts/make_hashes.py hunter2 dragon 123456 -a sha1 --show
```

You can also do it directly with Python — handy to understand what the tool is
comparing against:

```bash
python -c "import hashlib; print(hashlib.sha256(b'letmein').hexdigest())"
```

Or from your shell's built-in tools:

```bash
printf '%s' 'password' | md5sum      # Linux
printf '%s' 'password' | md5         # macOS
printf '%s' 'password' | shasum -a 256
```

> Note: `echo` adds a trailing newline, which changes the hash. Use
> `printf '%s'` (no newline) so the digest matches what the cracker computes.

## Usage examples

All of these were run against hashes generated by `make_hashes.py`.

### Dictionary attack (algorithm auto-detected)

```bash
H=$(python scripts/make_hashes.py password -a md5)
python -m hashcracker "$H" -w examples/wordlist.txt
```
```
[*] target : 5f4dcc3b5aa765d61d8327deb882cf99
[*] algo   : md5 (auto-detected)
[*] mode   : dictionary
              2 tried  |     7.1 H/s  |     0.3s  |    6.7%  |  ETA    4.0s
[+] password found: 'password'
password
```

The cracked password is printed on **stdout**; everything else (progress,
status) goes to **stderr**, so you can capture just the result:

```bash
pw=$(python -m hashcracker "$H" -w examples/wordlist.txt -q)
echo "recovered: $pw"
```

### Brute-force attack

```bash
# Force SHA-256, try all lowercase strings up to 3 chars
H=$(python scripts/make_hashes.py cab -a sha256)
python -m hashcracker "$H" -m brute --charset lower --max-length 3 -a sha256
```
```
[+] password found: 'cab'
cab
```

`--charset` accepts a **preset name** or a **literal string**:

| preset      | characters            |
|-------------|-----------------------|
| `digits`    | `0-9`                 |
| `lower`     | `a-z`                 |
| `upper`     | `A-Z`                 |
| `alpha`     | `a-zA-Z`              |
| `alnum`     | `a-zA-Z0-9` (default) |
| `printable` | letters+digits+punctuation |

```bash
# Literal charset: only these characters
python -m hashcracker "$H" -m brute --charset "abc" --max-length 4

# A 4-digit PIN
python -m hashcracker "$PIN_HASH" -m brute --charset digits --min-length 4 --max-length 4
```

### Controlling parallelism

```bash
# Use all cores (default), or pin a specific number
python -m hashcracker "$H" -m brute --charset lower --max-length 5 -j 8

# Single process (simplest; good for tiny jobs and debugging)
python -m hashcracker "$H" -m brute --charset lower --max-length 5 -j 1
```

### Full option list

```
positional:
  hash                  the target hash (hex digest) to crack

options:
  -m, --mode {dictionary,brute}   attack mode (default: dictionary)
  -a, --algo {auto,md5,sha1,sha256}  algorithm; 'auto' detects from length

dictionary mode:
  -w, --wordlist PATH   newline-separated wordlist file
  --no-count            skip the line-counting pre-pass (faster start, no %/ETA)

brute-force mode:
  --charset CHARSET     preset name or literal characters (default: alnum)
  --min-length N        minimum length (default: 1)
  --max-length N        maximum length (default: 4)

execution:
  -j, --workers N       worker processes (default: all cores; 1 = no multiprocessing)
  --batch-size N        candidates per task sent to a worker (default: 10000)
  -q, --quiet           suppress the live progress line
```

**Exit codes:** `0` found · `1` exhausted without a match · `2` usage/input
error · `130` interrupted (Ctrl-C).

## How it works (the design, briefly)

The central idea: **an attack is just a stream of candidate strings plus a
target digest.** Dictionary and brute-force modes differ only in *where the
candidates come from* — a file, or `itertools.product`. So the code splits
cleanly:

- `candidates.py` produces a **lazy iterator** of guesses (never materialising a
  huge space in memory).
- `engine.py` consumes that iterator, hashes candidates **in parallel**, and
  stops on the first match.

A few decisions worth understanding:

- **Multiprocessing, not threads.** Hashing is CPU-bound and Python's GIL stops
  threads from using multiple cores for Python code. Separate processes each get
  their own interpreter.
- **Batching.** Candidates are shipped to workers in batches of ~10,000. Sending
  one at a time would make inter-process communication cost more than the hash
  itself — batching is what makes it fast.
- **Digest-byte comparison.** The hot loop compares raw `digest()` bytes against
  the target (converted once with `bytes.fromhex`), skipping a hex-encode per
  guess.
- **`spawn`-safe shutdown.** On macOS/Windows workers are *spawned* (fresh
  processes), so the shared "stop now" flag is a `multiprocessing.Manager` event
  (a plain `Event` can't cross the spawn boundary). `imap_unordered` streams
  results back so we can stop the instant any worker wins.

### Performance note (an honest one)

On an 8-core machine, brute-forcing ~12.4M MD5 candidates took **8.4s with one
worker** vs **2.6s with eight** — about a **3.2× speedup**, not 8×. Why not
linear? The *main* process still generates every candidate and pickles each
batch before a worker can hash it; that serial work is a real bottleneck
(Amdahl's law in miniature). A tool optimised purely for speed would have each
worker generate its own slice of the space to avoid shipping candidates around —
a good exercise, but it would obscure the clean "one generator, one engine"
design this project is built to teach.

## Why this matters: the security lesson

This tool works *disturbingly well* against MD5/SHA-1/SHA-256 — and that is
exactly the point. Those are **general-purpose hash functions designed to be
fast**. Fast is the *opposite* of what you want for storing passwords, because
fast means an attacker can try billions of guesses per second.

Real systems defend against tools like this one:

1. **Salting.** A unique random salt per password means identical passwords get
   different hashes, which defeats precomputed **rainbow tables** and stops an
   attacker from cracking many accounts at once with a single pass.
2. **Slow, deliberately expensive hashing.** Password hashing functions like
   **bcrypt**, **scrypt**, and **Argon2** are engineered to be *slow* and (for
   scrypt/Argon2) **memory-hard**, so each guess costs meaningful time and RAM.
   This is called **key stretching / work factor** — you tune the cost so a
   legitimate login takes ~250ms but a brute-force attack becomes economically
   hopeless.
3. **Peppering, rate limits, MFA, breach monitoring** add further layers.

**The takeaways:**

- Never store passwords with plain MD5/SHA-1/SHA-256 (salted or not). Use
  **Argon2id**, **scrypt**, or **bcrypt** with an appropriate work factor.
- The strength of a password against brute force grows with **length and
  character-set size** — watch how much longer `--max-length 6` takes than `4`.
- Dictionary attacks win because people reuse common passwords. Length + true
  randomness beats cleverness.

If, after playing with this, you understand *why* the defences above exist, it's
done its job.
